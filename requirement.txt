# Agentic Data Quality & Observability Platform — Build Specification

You are a principal-level AI Engineer, Data Architect, and Data Quality Platform Engineer.

Build a **production-grade, enterprise-ready Agentic Data Quality (DQ) and Observability Platform** using the Claude API, Google Cloud Platform, and modern data engineering tooling. Every component must be fully implemented, connected, and executable — no stubs, no toy code.

---

## Primary Objective

Deliver an AI-powered Data Quality & Observability Agent that:

- Accepts BigQuery datasets and tables as input
- Discovers metadata via BigQuery `INFORMATION_SCHEMA` and optional Dataplex lineage
- Automatically generates technical DQ rules, business DQ rules, monitoring rules, freshness checks, schema drift checks, and referential integrity checks
- Uses Claude API reasoning to infer intelligent business rules, recommend thresholds, detect anomalies, and dynamically generate BigQuery SQL
- Persists DQ results to standardized BigQuery output tables
- Integrates natively with Airflow / Cloud Composer
- Emits alerts and structured observability signals
- Produces BI-consumable reporting views (Looker, Looker Studio, Power BI, Tableau)
- Enforces human-in-the-loop approval checkpoints before critical transitions
- Operates as a stateful agentic workflow with memory, tool use, retry handling, and orchestration

---

## Agent Architecture

Implement a modular multi-agent system. Each agent has a single responsibility and communicates through a shared state layer.

| Agent | Responsibility |
|---|---|
| **Orchestrator Agent** | Coordinates stage sequencing, manages state, handles retries and failures |
| **Metadata Discovery Agent** | Queries `INFORMATION_SCHEMA`, profiles columns, infers semantics |
| **Technical Rule Engine Agent** | Generates standard DQ rules (completeness, uniqueness, validity, etc.) |
| **Business Rule Recommendation Agent** | Uses Claude reasoning to infer business-specific rules from column semantics |
| **SQL Generation Agent** | Produces parameterized, idempotent, retry-safe BigQuery SQL |
| **Validation Agent** | Executes generated SQL and captures pass/fail results |
| **Monitoring Agent** | Tracks SLAs, freshness, runtime, anomalies, and health scores |
| **DAG Integration Agent** | Injects DQ tasks into existing Airflow DAGs or generates standalone DAGs |
| **Reporting Agent** | Creates BigQuery views and BI-ready aggregations |
| **Alerting Agent** | Routes failures to Slack and email with severity-aware escalation |
| **Human Approval Agent** | Manages approval checkpoints, persists decisions, maintains audit trail |

The orchestration layer must support:

- Multi-step reasoning with context carryover
- Tool calling (BigQuery, Dataplex, Airflow, alerting APIs)
- Persistent memory and state management
- Retry and circuit-breaker patterns
- Async and parallel rule generation
- Workflow resume and rollback

## Agent Workflow — Stage-by-Stage Implementation

Implement each stage sequentially. Do not skip stages or approval gates.

---

### Stage 1 — Scope & Metadata Resolution

**Input:** `project_id`, `dataset_id`, `table_names[]`

**Actions:**
- Query `INFORMATION_SCHEMA.COLUMNS`, `INFORMATION_SCHEMA.TABLE_OPTIONS`, `INFORMATION_SCHEMA.PARTITIONS`
- Retrieve: column names, data types, nullability, partition/clustering keys, row counts, PK/FK candidates
- Optionally query Dataplex for tags, sensitivity classifications, and lineage
- Fall back to `INFORMATION_SCHEMA` if Dataplex is unavailable
- Run column-level profiling: distinct counts, null rates, min/max, sample values, cardinality
- Infer column semantics using Claude (e.g., detect email, phone, currency, date, status fields)

**Output:**
- `metadata_summary.json` — structured table and column metadata
- `profiling_report.json` — per-column statistical profile
- `semantic_inference.json` — Claude-inferred business types per column

---

### Stage 2 — Technical DQ Rule Generation

Automatically generate standard rules across the following categories. For every rule, assign: `rule_id`, `severity`, `threshold`, `execution_frequency`, and executable BigQuery SQL.

| Category | Rules |
|---|---|
| **Completeness** | Not-null checks, mandatory field enforcement, sparse data detection |
| **Uniqueness** | Duplicate detection, composite PK uniqueness |
| **Validity** | Data type validation, regex conformity, allowed values / enum checks |
| **Integrity** | Referential integrity, cross-table consistency |
| **Freshness** | Late arrival detection, SLA breach checks, timestamp lag |
| **Volume** | Row count anomalies, sudden drops/spikes vs. baseline |
| **Schema Drift** | Column additions/removals, data type changes |
| **Consistency** | Cross-column validation, derived field correctness |

---

### Stage 3 — Business Rule Recommendation (Claude Reasoning)

Use Claude to infer intelligent business rules from column semantics and profiling data.

**Examples of inferred rules:**
- `email` columns → RFC 5322 regex validation
- `status` columns → allowed values enforcement
- `amount` / `revenue` columns → positive range checks, outlier detection
- `date` columns → future/past boundary constraints
- `country` / `state` columns → reference lookup conformity
- `PII` columns → masking / anonymization verification

**Claude must:**
- Use chain-of-thought reasoning to justify each recommendation
- Output structured JSON with rule metadata and SQL
- Assign severity and recommended thresholds
- Suggest anomaly detection strategies for numeric and temporal columns

Implement versioned prompt templates for: rule inference, SQL generation, severity classification, and anomaly strategy selection.

---

### Stage 4 — Business Rule Elicitation

Present recommended rules to the user interactively.

**Allow the user to:**
- Accept, modify, or reject any recommended rule
- Add custom business rules
- Set severity: `INFO | WARN | FAIL`
- Set custom thresholds and execution schedules

**Persist approved rules to:**
- YAML rule files (human-readable, version-controlled)
- JSON rule files (machine-consumable)
- BigQuery `dq_rule_config` table (runtime lookup)

---

### Stage 5 — Checkpoint 1: Human Approval (Rule Set)

**Before SQL generation**, enforce a full approval gate.

- Display all rules with metadata (severity, threshold, schedule, SQL preview)
- Require explicit approval
- Support in-place edits before approval
- Version the approved rule set with a `rule_set_version_id`
- Write approval decision, approver identity, and timestamp to `dq_audit_log`

**Do not proceed to Stage 6 without recorded approval.**

---

### Stage 6 — DQ SQL Generation

Generate executable BigQuery SQL for every approved rule.

**Requirements:**
- Parameterized queries (no hardcoded values)
- Idempotent execution (safe to re-run)
- Retry-safe design
- Full exception handling and structured logging
- Every execution writes results to the standardized `dq_results` table

**Supported output formats:**
- Pure BigQuery SQL
- BigQuery Stored Procedures
- dbt test YAML
- Dataplex Auto DQ configuration YAML

**`dq_results` table schema:**

```sql
CREATE TABLE IF NOT EXISTS `{project}.{dataset}.dq_results` (
    run_id              STRING       NOT NULL,
    rule_id             STRING       NOT NULL,
    project_id          STRING       NOT NULL,
    dataset_name        STRING       NOT NULL,
    table_name          STRING       NOT NULL,
    column_name         STRING,
    rule_type           STRING       NOT NULL,
    severity            STRING       NOT NULL,  -- INFO | WARN | FAIL
    status              STRING       NOT NULL,  -- PASS | FAIL | ERROR | SKIPPED
    observed_value      STRING,
    expected_value      STRING,
    threshold_value     STRING,
    failure_count       INT64,
    execution_time      TIMESTAMP    NOT NULL,
    execution_duration_seconds FLOAT64,
    query_executed      STRING,
    created_at          TIMESTAMP    NOT NULL    DEFAULT CURRENT_TIMESTAMP()
)
PARTITION BY DATE(execution_time)
CLUSTER BY table_name, rule_type, severity;
```

**If NO:**

**Generate:**

- Deployment instructions for Cloud Composer

### Stage 8 — Monitoring & Observability

Implement enterprise-grade observability.

**Monitoring capabilities:**

**Alerting:**

**Instrumentation:**

Present:
- Monitoring schedules per rule

Require explicit approval. Persist config and approval to `dq_audit_log`.

**Do not activate monitoring or alerting without recorded approval.**

Generate BigQuery views and materialized views for BI consumption.

**Required views:**

|---|---|

| `v_dq_table_health` | Per-table health score, last run time, open failures |

**All views must:**

- Expose standard metrics: pass percentage, failure count, critical failure count, mean freshness lag, health score, daily trend

## Claude API Implementation

- **Model:** `claude-sonnet-4-6` (default), with upgrade path to Opus for complex reasoning tasks
- **Features required:** streaming responses, tool use, structured JSON output, prompt caching
- **Reliability:** exponential backoff retry logic, rate limit handling, token budget management
- **Prompt architecture:** versioned system prompts and task prompts per agent, stored in `prompts/`
- **Context management:** sliding window summarization for long workflows, state compression between stages
