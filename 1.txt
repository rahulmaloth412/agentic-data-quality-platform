# 2. Rule Generation Toggle Behavior

In the “New DQ Workflow” section there are currently two toggles:

* Technical Rules
* Business Rules

Enhance the behavior as follows:

## Business Rules Toggle Logic

* If the user disables/unselects “Generate Business Rules”:

  * The engine must NOT invoke any business-rule generation logic.
  * The AI agent must NOT recommend, infer, or create business rules.
  * Only technical DQ rules should be generated and processed.

## Technical Rules Toggle Logic

* If technical rules are enabled:

  * Only technical metadata-driven rules should be generated.
  * Examples:

    * Null checks
    * Duplicate checks
    * Datatype validation
    * Referential integrity
    * Length validation
    * Regex validation
    * Freshness checks

The backend orchestration should strictly respect the selected toggles.

# 3. Rules Manager Refactor

Refactor the “Rules Manager” section into a more enterprise-grade approval workflow.

## Separate Rule Categories

After rules are generated, display them in two clearly separated sections:

### Technical DQ Rules

Examples:

* Null validation
* Duplicate detection
* Datatype checks
* Range validation

### Business DQ Rules

Examples:

* Revenue cannot be negative
* Order date must be before shipment date
* Active customer must have active subscription

The UI should visually distinguish these sections with:

* Separate containers/cards
* Headers
* Different icons or labels
* Clear categorization

# 4. Rule-Level Actions

For every generated rule provide the following actions:

## Select / Unselect

* Checkbox or toggle to approve/reject the rule for final SQL generation.
* Only selected rules should be included in downstream processing.

## Delete

* Allow users to permanently remove generated rules from the session/workflow.

## Approval Workflow

* SQL generation should happen ONLY after rule approval.
* Unselected or deleted rules must NOT be included in:

  * SQL generation
  * Stored procedure generation
  * Execution pipelines
